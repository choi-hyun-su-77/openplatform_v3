# 개발자 매뉴얼 + 스캐폴드/화면형태/메뉴등록 작업 지침서 작성 프롬프트
# (코드베이스 분석 기반 — 기술 스택 독립형)

## 역할 & 목표
당신은 시니어 풀스택 아키텍트이자 테크니컬 라이터입니다.
산출물은 **사람 개발자가 직접 손으로 따라 작업하기 위한 지침서**입니다.

이 프롬프트에는 **특정 프레임워크·라이브러리·파일명·디렉토리 구조·패턴 개수·화면 형태 개수·메뉴 등록 단계 수에 대한 어떤 가정도 들어 있지 않습니다.** 모든 구체적 사실은 Phase 0에서 **코드베이스 내 기존 문서와 실제 소스코드 분석을 통해 도출**합니다. 도출 결과를 바탕으로 표준 골격에 본문을 채워 넣는 방식으로 진행합니다.

각 SOP는 다음 질문에 명확히 답해야 합니다:
- 어느 파일을 / 어디서 / 어디로 / **몇 개를 복사**하는가?
- 어느 파일을 / 어떤 위치에 / 어떤 이름으로 **새로 만드는가**?
- 어느 파일의 / 어느 줄(섹션)을 / 어떻게 **수정하는가**?
- 도메인명·서비스명을 / **어떤 식별자들로 치환**해야 하는가? (전수 치환표)
- 메뉴를 / **어떤 등록 지점들**(코드베이스에서 발견된 N개)에 등록하는가?
- 검증은 / 어떤 동작이 보이면 성공인가?

## 핵심 원칙
1. **사람용 산출물**: 모든 단계는 개발자가 IDE에서 수행하는 동작 단위. "AI가 알아서 한다" 표현 금지.
2. **1차 정보원 우선**: 코드베이스 내 기존 산출물(README, ADR, 가이드, 설계 문서, 주석 등)과 실제 소스코드만 근거로 사용. 출처 없는 추측·창작 금지. 모든 인용은 `[code: 경로:Line]` 또는 `[doc: 경로]` 형식.
3. **기술 스택 독립**: 프롬프트 본문에 특정 프레임워크/라이브러리/명명 가정을 박지 않는다. Phase 0에서 발견한 사실로만 본문을 채운다.
4. **표 형식 강제**: 복사/신규/수정/치환은 반드시 표. 산문 서술만 있는 단계는 게이트 실패.
5. **레퍼런스 모범 워크스루**: 각 SOP는 코드베이스에서 발견한 실제 모범 사례 1개를 처음부터 끝까지 따라가는 워크스루로 마무리.
6. **재현 가능성**: 같은 SOP를 두 명의 개발자가 따라도 동일한 결과.
7. **추상 표현 금지**: `적절히`, `필요 시`, `상황에 따라` 등 모호 표현 검출 시 구체화.

### Phase 2 — 백엔드 스캐폴드 SOP 작성 (병렬 디스패치, **개수는 Phase 0.C 결과**)

> Phase 0.C에서 도출된 패턴 개수만큼 SOP 파일을 만든다. SOP들은 **동일한 단계 헤더**를 글자 단위로 공유하되, 본문은 패턴별 구체화.

#### 2.0 결정 트리 — `scaffolds/00_decision_tree.md`
- [ ] Mermaid flowchart로 작성 — Phase 0.C.5의 결정 요인을 분기점으로
- [ ] 모든 분기는 정확히 하나의 패턴 SOP에 도달, 막다른 분기 0
- [ ] 패턴이 1개뿐이면 결정 트리 대신 "단일 패턴 적용 안내"

#### 2.1–2.N 패턴별 SOP — `scaffolds/{NN}_{pattern_name}.md`

> N = Phase 0.C에서 도출된 패턴 수. 각 SOP는 표준 단계 골격을 따른다.

**표준 단계 골격** (모든 백엔드 패턴 SOP 공통, 본문은 패턴별 구체화):
```
[Step 1]  사전 결정 — 도메인 식별자(0.E 변환규칙 적용), 화면 형태 결정, 권한 결정
[Step 2]  영속 계층 — DDL/스키마/마이그레이션
[Step 3]  메뉴·권한 메타데이터 등록 → menu/menu_registration.md 참조
[Step 4]  데이터 액세스 계층
[Step 5]  비즈니스 로직
[Step 6]  진입점 / 라우팅
[Step 7]  표준 응답·로깅 컨벤션 적용 (Phase 0.H 기반)
[Step 8]  화면 작성 → screens/{형태}.md 참조
[Step 9]  클라이언트 라우터 / 메뉴 매핑
[Step 10] 테스트 (단위·통합·E2E 중 코드베이스 컨벤션에 맞는 것)
[Step 11] 자기검증 (Phase 0.H 컨벤션 체크리스트)
[Step 12] PR 준비
```

> 특정 패턴에서 어떤 Step이 의미 없으면(예: 외부 시스템에 영속 계층이 없는 경우) **"해당 없음"** 으로 명시하고 사유를 적는다. 단계 번호 자체는 유지.

**각 단계마다 의무 4표** (표 누락 시 게이트 실패):
- 표 1. 복사할 파일 — 원본 → 복사 후 → 치환 식별자 → 비고
- 표 2. 신규 생성할 파일 — 경로 / 역할 / 골격 출처(`templates/...`)
- 표 3. 수정할 기존 파일 — 경로 / 수정 위치(섹션·앵커) / 추가·변경 요약
- 표 4. 식별자 치환 — From → To (적용 범위)

**각 SOP 마지막**: Phase 0.I에서 선정한 모범 도메인을 처음부터 끝까지 따라가는 1페이지 워크스루.

#### 2.X 빈 골격 템플릿

**표준 7섹션 골격**:
```

6. 부모-자식·라우터 연동 — 다이얼로그/패널 간 통신 패턴
7. 모범 워크스루       — Phase 0.I 모범 화면 따라가기
```

## 누락 방지 게이트
1. 모든 단계가 **표 형식** 4표(복사/신규/수정/치환) 의무. 산문만 있는 단계 실패.
2. 모든 SOP에 **레퍼런스 모범 워크스루**가 마지막에 붙어 있어야 함.
3. 메뉴 등록 절차 서술은 `menu/menu_registration.md` **단 한 곳**.
4. 결정 트리 모든 분기가 정확히 하나의 SOP에 도달.
5. 빈 템플릿은 식별자 치환 후 즉시 빌드/렌더 가능.
6. SOP 단계/섹션 헤더 글자 일치.
7. 추상 표현 0건.
8. 모든 사실 주장에 `[code: ...]` / `[doc: ...]` 출처 표기.
9. **본 산출물 어디에도 Phase 0에서 발견되지 않은 기술/명명/구조 가정이 등장하지 않을 것** (가장 중요한 게이트).

## 진행 보고 규칙
- 매 Phase 완료 시 `info.md`에 누적 append.
- 차단/모호 발생 시 `warn.md` 또는 `fatal.md`(사용자 확인 필요)에 기록 후 즉시 보고.
- Phase 0 완료 시점에 **1회 의무 보고** (발견된 패턴/화면/메뉴 단계 수, 갭 항목, 후속 Phase 진행 가부 문의).
- 최종 한 줄 요약: `✅ HANDBOOK + N backend SOPs + M screen SOPs + menu({K}-step) SOP + recipes + templates, 드라이런 통과`
